Coding errors/bugs

  • A coding language and its functionality are governed by its rules and restrictions
    • e.g. cannot add 5 and “g” together, wrong set of brackets to define function
  • Works in unison with the systems it works with
    • e.g. computer, operating system, internet
  • Bugs/errors will happen to users when they run through a program
    • Within code: we need a mechanism to stop the process and alert users of issues

Programming errors

  • Exist when process doesn’t work due to some aspect of the programming
    • Syntax, logical, runtime, time limit errors

Syntax errors

  • Code violates rules of programming
    • Most common error
    • e.g. [], casing, spelling
x <- mtcars$mpg
mean(x)
[1] 20.09062
maen(x)
Error in maen(x): could not find function "maen"

Logical errors

  • Incorrect logic used within code
    • Typically in association with &, |, T, F
    • Can extend to how we would operate through a process
oddEven <- function(x){
  if (x %% 2 == 0){
    print("odd")
  } else {
    print("even")
  }
}
oddEven(2)
[1] "odd"

Runtime errors

  • Unexpected condition encountered when running through code
data <- NA
data[2, ]
Error in data[2, ]: incorrect number of dimensions

How R deals with errors

  • All these errors/problems get funneled into three different conditions
    • Errors: stops code execution when it occurs
      • Error message is returned to user via stop()
    • Warnings: sends message to user when code does not run as intended
      • Warning message is returned to user via warning()
    • Messages: purely informational
      • message()

Debugging errors (review)

Examples:

  • Error in madian(c(1, 7, 13)) : could not find function "madian"
  • Error in rnorm() : argument "n" is missing, with no default

Key thing here is to read through the first part of error message, and then resolve.

Common solutions to errors

  • Read through the error!!
    • Consider processes around location of error (e.g. spelling, functionality)
  • Making sure everything is spelled correctly and has the right case
    • R is case-sensitive!
  • All “(” must pair with a “)”
  • Break apart code, assigning actions to objects as you go along

Implementing your own errors/warnings

  • To throw an error: use the stop() function in conjunction with an if() statement
    • Can capture unwanted behavior
      • e.g. wrong object type, limited functionality
    • tryCatch() is the more formal way; we will not focus our discussion on this

Example: Just doubles, please

oddEven <- function(x){
  if (!is.double(x)){
    stop("Value is not of a double type.")
  }
  if (x %% 2 == 0){
    print("odd")
  } else {
    print("even")
  }
}
oddEven("apple")
Error in oddEven("apple"): Value is not of a double type.
