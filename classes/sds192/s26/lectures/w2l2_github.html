Working directories

  • Important to set your working directory
    • If you create a project in directory, your working directory is set
    • Otherwise, use setwd()
      • Need to find file path (Properties for Windows users, Cmd + Opt + C for Mac users on file)
setwd("..") # to go back a folder
setwd("C:/Users/jericho/Documents")

R projects

  • Associated with an R working directory
    • Results in own workspace and history
  • Can be associated with a version control repository (e.g. Git)
    • Your lab submission repository is an example
  • To make: File > New Project

Data imports

  • Can be seen as a .txt or .csv file
    • read.table() for .txt files
    • read.csv() for .csv files
  • Specify file path and if the header exists

A .csv file

  • A comma-separated file
    • Rows are created with new lines (e.g. return/enter)
    • Columns are separated by commas

Importing data from a .csv file

  • Need to place data into same working directory as your code
    • Adjust file path as necessary
nba_data <- read.csv("nba2024.csv", header = T)

Absolute paths

  • Absolute paths list all folders from your root folder to the location of the file
    • Avoid this as much as possible!
    • Reproducibility issues

Example: “C:/Users/jericho/Documents/SDS 192”

Relative paths

  • Relative paths direct to the desired file from your current location
    • Move up a directory: ../
    • Move up two directories: ../../
    • Move up a directory and into datasets folders: ../datasets/

Example:

  • datasets/
  • code.qmd
  • README.md

Importing tips

  • Data should be rectangular!
    • No merged columns/rows
    • Values in every cell
  • Avoid Excel and other proprietary formats
  • Avoid spaces in column names if possible

Repairing names

  • If spaces exist in column name, you must use backticks in order to reference column
    • e.g. `column name`
    • Other solutions:
      • Assign make.names(names(your_data)) to names(your_data)
      • Fix each unwanted column name by accessing names(your_data)

Missing data

  • In almost all cases, missing data will occur
    • May show up as blank or NA
  • Solutions:
    • Use na.omit(your_data) to remove observations with at least one missing value
      • Easy solution, but may be problematic if missing values occur in various spots in data
    • Imputation (only when data are missing at random)
      • Use of replace() function

Example

cities <- c("Northampton", "Amherst", "Hadley")
temps <- c(57, 60, NA)
inter <- c(T, F, F)
data <- data.frame(city = cities, temp = temps, inter = inter)
na.omit(data)
         city temp inter
1 Northampton   57  TRUE
2     Amherst   60 FALSE
data$temp <- replace(data$temp, is.na(data$temp), mean(data$temp, na.rm = T))
data$temp
[1] 57.0 60.0 58.5

Unnecessary data

  • Suppose there are small pockets of data that you deem are not the main focus of your project
    • Use of subsetting
    • To remove: Use !(condition) within []
    • To keep: Use condition within []
mtcars[mtcars$mpg < 15, ]
mtcars[!(mtcars$mpg < 15), ]

Version Control

Version control

  • Allows us to save and refer to versions of digital documents over time
  • Supports “reproducibility” by allowing us to retrace our steps in a systematic way
    • e.g. a resume folder
    • e.g. a package

Question: Why is version control important?

Git

Git

  • Open source command line tool for version control
  • Installed on your local machine (OS-specific)
  • Does not require the use of any external tools

GitHub

  • Web-hosting platform for git projects
  • Hosts repositories for publishing/sharing work with the broader coding community

Creation of a repository

  • Multiple ways to do this:
    • Create repository on GitHub, then clone on your computer using RStudio/GitHub Desktop
    • Create new project on RStudio, then connect to GitHub

*Note:** The repository will be seen locally and remotely.

GitHub workflow

  1. Pull changes from GitHub (remote) to your machine (local)
    • Important to ensure there are not any conflicts!
  2. Make changes on your computer
  3. Commit changes
    • Do this often and leave descriptive commit messages!
  4. Pull changes from remote again if working with others.
  5. Push changes from machine (local) to GitHub (remote)

Pulls

  • Retrieves any changes made in remote repository, adds those changes to machine
    • Needs to be done when collaborating with multiple people
    • When not done, merging conflicts can arise

Example: pull

Commits

  • Commits are saves to a digital document
    • Stored snapshot of the document at that time
    • Includes a unique hash that allows you to track the history of changes to the document
  • When you commit, you create a commit message where you document your changes.
    • Commit messages should be descriptive of the changes made

Differences and pushing

  • On committing, we can review the differences between the previous and current version of a document
    • Additions highlighted in green
    • Deletions highlighted in red
  • Commit, pull, and push

Example: differences and pushing

Why GitHub?

  • Reproducibility
  • Collaboration
  • Licensing
  • Robust way to add descriptions (e.g. README files) and issues

GitHub vocabulary

  • Fork: Create a copy of a GitHub repository for editing in your own user account
  • Clone: Download a copy of a remote repository to your local machine
  • Pull: Fetch changes to code from a remote repository to local machine
  • Stage: Select the files you wish to commit changes on
  • Diff: Review the differences between different versions of files
  • Commit: Take a snapshot of changes to files
  • Push: Upload changes to code from local machine to remote repository

Collaboration on GitHub

  • GitHub assigns permissions to different users to manage and make changes to files stored in a GitHub repository
    • Use of branches and pull requests
  • Separate users can clone a repo to their local machine to make changes at the same time
    • If cloned, users will not know what changes the other is making!
    • This is why pulling often is so important!
    • There may be a different version of a file in GitHub by the time you go to push your changes

Push error

This error indicates that you don’t have the latest version of the files on your local machine. You should pull, then push again.

Pull error

This error indicates that pulling the changes that you or someone else made in remote would overwrite the changes you made on your local machine. You should commit, pull, and then push.

Collaboration challenges

  • What happens when two users have cloned a GitHub repo to their local machine and are making changes to that repo in the same place at the same time?
  • When pushing you will get an error indicating that there is a merge conflict.
  • git wants to know which version of the file you prefer - yours or your collaborators.
  • You have to manually resolve these conflicts in an editor.

Merge conflicts

Delete lines so that only desired code appears. Commit changes, and then push.
