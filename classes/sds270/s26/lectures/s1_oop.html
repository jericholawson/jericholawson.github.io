Introduction

Extending (and clarifying) our conversation on classes in R, I’ll walk through the example shown in class and show how it can be incorporated in R code.

A Class

A class (using the S3 system) can be defined in a few ways: either formally through the use of helpers or as we go by altering an object’s class attribute.

While there are ways to formally set the object’s class to a new class, we can do so (and get away with this) by adding a character (named as the class) to its class attribute. To do this, we can use:

class(my_object) <- c("my_class", class(my_object))

We are most definitely prone to errors when trying to associate an object as a new class. We will talk in Monday’s lecture after the break about ways we can let the user create the object (associated with the class) in a more formal manner.

The Car, Bicycle, and Vehicle Classes

If we dichotomize each of these three objects, cars and bicycle are their own categories and they share components that would make them vehicles. As such, we would want to design a system in which the vehicle class is a parent class of the car and bicycle classes.

For each of these classes, we discussed actions and attributes that define these classes. For a vehicle object, we may have the following:

  • Type of vehicle
  • Color
  • Can be used to travel to some destination in a certain number of hours

Let’s create an object that houses the first two qualities:

myVehicle <- list(type = "car", color = "blue")

Quite simply, we can make a list with these qualities. In next Monday’s class, we’ll discuss helpers and constructors we can create to allow the user to make these objects easily.

To associate this myVehicle object of the vehicle class, we can add the class name to the class attribute:

class(myVehicle) <- c("vehicle", class(myVehicle))
class(myVehicle)
[1] "vehicle" "list"

Alternatively, we can also use the structure() function to do this, where we can put the object as the first argument and the class name as the second argument:

myVehicle <- structure(myVehicle, class = "vehicle")

The Methods

For any method, we need to have a generic, default, and class-specific function. Why? With the S3 class system, the generic’s job is to perform method dispatch. If I use the print() function, R needs a way to determine how to access the class-specific functions.

As such, suppose we create the travel function for our vehicle class. We start with the generic, which will be written as:

travel <- function(x, ...){
  UseMethod("travel")
}

In this specific function only, we use the UseMethod() function on the name of the method we’ll be creating. The dots now have a purpose here: carry any potential arguments through a search path of possible travel() methods depending on the classes associated with an object.

However, if I run the travel() function on our object, I get this error:

travel(myVehicle)
Error in `UseMethod()`:
! no applicable method for 'travel' applied to an object of class "c('vehicle', 'list')"

As the error states, there is no travel() method on the classes associated with our object. This is where a method for the default class is needed. The default class is a pseudo-class – it serves as a fallback in case we do not have a method that exists for the class we are working with.

The default function can be used to run an operation on the object (regardless of its class or type), or it can serve to tell the user that a method doesn’t exist for the classes associated with the object. For example:

travel.default <- function(x, ...){
  print(NULL)
}

Here, I just printed NULL. So, when I run the travel() function once again:

travel(myVehicle)
NULL

I get the NULL statement since I do not have a travel() method for the vehicle class. Now, I can add in this travel() method. When you write the function, you will need to have it named as methodName.className. This will allow method dispatch to correctly pick the appropriate method. As an example:

travel.vehicle <- function(x, dest, ...){
  if (dest == "Amherst"){
    print("6 miles")
  } else {
    print(NA)
  }
}

Now, if I run the travel() function on myVehicle:

travel(myVehicle, "Amherst")
[1] "6 miles"

I’ve now created a method for my vehicle class, with proper structure needed for method dispatch and missing methods.

Inheritance

Inheritance refers to attributes and methods that can be shared among classes. This allows us to define parent and child classes. Child classes will have its methods called first in method dispatch. An object of a child class will take on all of the behavior of the parent class, but with specifics defined in the child class. As such, when you view your class attribute, the child class will come first.

In our example, let’s associate myVehicle with the car class.

class(myVehicle) <- c("car", class(myVehicle))

If we print out the class attribute:

class(myVehicle)
[1] "car"     "vehicle" "list"

The car class will be the child class of the vehicle class. A car should take on all of the components and actions of a vehicle, but with more specifics that make a car a car.

To emphasize this, we will create a travel() method for the car class. Perhaps, we now want to specify the time it takes to get to a destination. Then,

travel.car <- function(x, dest, ...){
  if (dest == "Amherst"){
    print("20 minutes")
  } else {
    print(NA)
  }
}

If we use the travel() function now:

travel(myVehicle, "Amherst")
[1] "20 minutes"

We’re now using the method corresponding to the car class. As you notice, we now have no output from the vehicle class; perhaps we still want this output. To do so, we can add a NextMethod() call to our method associated with the car class – this will print the same output as before, but then use the method thereafter from the parent class and run the actions from that method. As such, if we do change the travel() method:

travel.car <- function(x, dest, ...){
  if (dest == "Amherst"){
    print("20 minutes")
  } else {
    print(NA)
  }
  NextMethod()
}

Then, we can get this output:

travel(myVehicle, "Amherst")
[1] "20 minutes"
[1] "6 miles"

This will be how object-oriented programming works in R, at least initially. We’ll go into some more specifics on method dispatch and helpers later.
